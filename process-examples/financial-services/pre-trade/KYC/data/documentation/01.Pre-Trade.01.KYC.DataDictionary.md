## Process Input Data

### Core Application Fields

#### applicationId

- **Type:** String
- **Format:** `APP-YYYYMMDD-XXXXXX` (where X is alphanumeric)
- **Required:** Yes
- **Description:** Unique identifier for the onboarding application
- **Example:** `APP-20250108-F7K9M2`
- **Validation:** Must match pattern `^APP-[0-9]{8}-[A-Z0-9]{6}$`

#### submissionTimestamp

- **Type:** DateTime
- **Format:** ISO 8601 (UTC)
- **Required:** Yes
- **Description:** When the application was submitted
- **Example:** `2025-01-08T14:32:18Z`
- **Usage:** Used for SLA calculations and audit trail

#### clientType

- **Type:** Enum
- **Values:** `individual`, `corporate`
- **Required:** Yes
- **Description:** Type of client being onboarded
- **Impact:** Determines required documents and verification processes

### Individual Applicant Fields

#### title

- **Type:** Enum
- **Values:** `Mr`, `Mrs`, `Ms`, `Miss`, `Dr`, `Prof`, `Other`
- **Required:** Yes (for individuals)
- **Description:** Individual's title

#### firstName

- **Type:** String
- **Length:** 1-100 characters
- **Required:** Yes (for individuals)
- **Description:** Individual's first name or given name(s)
- **Validation:** UTF-8 characters, no numbers

#### lastName

- **Type:** String
- **Length:** 1-100 characters
- **Required:** Yes (for individuals)
- **Description:** Individual's last name or surname
- **Validation:** UTF-8 characters, no numbers

#### dateOfBirth

- **Type:** Date
- **Format:** ISO 8601 date (YYYY-MM-DD)
- **Required:** Yes (for individuals)
- **Description:** Individual's date of birth
- **Example:** `1985-06-15`
- **Validation:** Must be 18+ years old
- **Usage:** Age verification, sanctions screening, biometric matching

#### nationality

- **Type:** String
- **Format:** ISO 3166-1 alpha-3 country code
- **Required:** Yes (for individuals)
- **Description:** Individual's nationality
- **Example:** `GBR`, `USA`, `FRA`
- **Validation:** Must be valid 3-letter country code
- **Usage:** Risk assessment, sanctions screening

#### taxResidency

- **Type:** Array of Strings
- **Format:** ISO 3166-1 alpha-3 country codes
- **Required:** Yes (for individuals)
- **Description:** Countries where individual is tax resident
- **Example:** `["GBR"]` or `["USA", "GBR"]`
- **Usage:** CRS/FATCA reporting, risk assessment

#### sourceOfWealth

- **Type:** Enum
- **Values:** `employment`, `business-ownership`, `inheritance`, `investments`, `property`, `other`
- **Required:** Yes (for individuals)
- **Description:** Primary source of individual's wealth
- **Usage:** Risk assessment, source of funds verification

### Corporate Applicant Fields

#### legalName

- **Type:** String
- **Length:** 1-200 characters
- **Required:** Yes (for corporates)
- **Description:** Full legal name of the entity
- **Example:** `MaedUp Technology Solutions Limited`

#### registrationNumber

- **Type:** String
- **Required:** Yes (for corporates)
- **Description:** Company registration number from jurisdiction of incorporation
- **Example:** `09876543`
- **Usage:** Entity verification, registry checks

#### incorporationCountry

- **Type:** String
- **Format:** ISO 3166-1 alpha-3 country code
- **Required:** Yes (for corporates)
- **Description:** Country of incorporation
- **Example:** `GBR`
- **Usage:** Jurisdiction risk assessment

#### lei

- **Type:** String
- **Format:** 20 alphanumeric characters
- **Required:** No (but recommended)
- **Description:** Legal Entity Identifier
- **Example:** `213800ABCDEF12345678`
- **Validation:** Must match pattern `^[A-Z0-9]{20}$`
- **Usage:** Entity verification, regulatory reporting

#### beneficialOwners

- **Type:** Array of Objects
- **Required:** Yes (for corporates)
- **Description:** Individuals with >25% ownership or control
- **Minimum Items:** 0 (but must explain if none)
- **Usage:** Ultimate beneficial owner identification, PEP screening

#### ownershipPercentage

- **Type:** Number
- **Range:** 0-100
- **Required:** Yes (for each beneficial owner)
- **Description:** Percentage of ownership or control
- **Example:** `55`
- **Usage:** Determining who requires screening

## Task Output Data

### Document Verification Output

#### documentsVerified

- **Type:** Boolean
- **Required:** Yes
- **Description:** Overall result of document verification - true if all documents pass
- **Values:** `true`, `false`
- **Usage:** Gateway decision - determines if process continues or requires remediation

#### overallConfidenceScore

- **Type:** Number
- **Range:** 0-100
- **Required:** Yes
- **Description:** Overall confidence score for verification
- **Example:** `94.5`
- **Usage:** Quality assessment, audit trail

#### verificationStatus

- **Type:** Enum
- **Values:** `verified`, `failed`, `manual-review-required`
- **Required:** Yes (for each document)
- **Description:** Verification outcome for individual document
- **Usage:** Determining overall verification result

### Sanctions Screening Output

#### sanctionsMatch

- **Type:** Boolean
- **Required:** Yes
- **Description:** Whether any sanctions matches were found
- **Values:** `true`, `false`
- **Usage:** Critical gateway decision - true blocks onboarding and triggers investigation

#### isPEP

- **Type:** Boolean
- **Required:** Yes
- **Description:** Whether individual or beneficial owners are politically exposed persons
- **Values:** `true`, `false`
- **Usage:** Risk assessment input, triggers enhanced due diligence

#### matchScore

- **Type:** Number
- **Range:** 0-100
- **Required:** Yes (for each match)
- **Description:** Confidence score for sanctions or PEP match
- **Example:** `94.7`
- **Threshold:** Typically 80+ requires investigation

#### pepType

- **Type:** Enum
- **Values:** `head-of-state`, `head-of-government`, `government-minister`, `senior-politician`, `senior-government-official`, `judicial-official`, `military-official`, `senior-executive-soe`, `political-party-official`, `rca`, `family-member`, `close-associate`
- **Required:** Yes (for PEP matches)
- **Description:** Classification of PEP type (rca = relative or close associate)
- **Usage:** Determines level of enhanced due diligence required

### Risk Assessment Output

#### riskCategory

- **Type:** Enum
- **Values:** `Low`, `Medium`, `High`
- **Required:** Yes
- **Description:** Overall risk category determined by DMN decision table
- **Usage:** Primary routing decision - determines approval path

#### overallRiskScore

- **Type:** Number
- **Range:** 0-100
- **Required:** Yes
- **Description:** Composite risk score calculated from multiple risk factors
- **Example:** `22.5`
- **Calculation:** Weighted average of individual risk factor scores

#### autoApprovalEligible

- **Type:** Boolean
- **Required:** Yes
- **Description:** Whether case is eligible for automated approval without manual review
- **Values:** `true`, `false`
- **Usage:** Determines if manual compliance review is required

#### approvalAuthority

- **Type:** Enum
- **Values:** `system`, `compliance-officer`, `senior-compliance`, `mlro`
- **Required:** Yes
- **Description:** Level of authority required for approval (MLRO = Money Laundering Reporting Officer)
- **Usage:** Task assignment, compliance governance

## Process Variables

Process variables are maintained throughout the process instance and used for gateway decisions.

#### clientId

- **Type:** String
- **Format:** `CLI-YYYYMMDD-XXXXXX`
- **Description:** Unique client identifier assigned after initial data capture
- **Example:** `CLI-20250108-000234`
- **Lifecycle:** Created in Capture Client Data task, used throughout

#### sanctionsCleared

- **Type:** Boolean
- **Description:** Result of compliance investigation after sanctions hit
- **Values:** `true`, `false`
- **Usage:** Gateway decision after compliance investigation
- **Note:** Only set if sanctionsMatch was true

#### approvalDecision

- **Type:** Enum
- **Values:** `approved`, `rejected`
- **Description:** Final decision from compliance review
- **Usage:** Gateway decision determining account opening or rejection

## Reference Data

#### riskScore

- **Type:** Number
- **Range:** 0-100
- **Description:** Numerical risk score for jurisdiction
- **Ranges:**
  - 0-25: Low risk
  - 26-50: Medium risk
  - 51-75: Medium-High risk
  - 76-95: High risk
  - 96-100: Prohibited

- **Description:** Whether country is FATF compliant
- **Values:** `true`, `false`
- **Usage:** Risk assessment input

#### corruptionPerceptionIndex

- **Type:** Number
- **Range:** 0-100
- **Description:** Transparency International corruption perception score (100 = least corrupt)
- **Source:** Transparency International
- **Usage:** Jurisdiction risk factor

## Data Types and Formats

### Standard Formats

#### ISO 8601 DateTime

- **Format:** `YYYY-MM-DDTHH:MM:SSZ`
- **Example:** `2025-01-08T14:32:18Z`
- **Timezone:** Always UTC (Z suffix)
- **Usage:** All timestamps

#### ISO 8601 Date

- **Format:** `YYYY-MM-DD`
- **Example:** `1985-06-15`
- **Usage:** Dates of birth, document issue/expiry dates

#### ISO 3166-1 Alpha-3

- **Format:** Three uppercase letters
- **Example:** `GBR`, `USA`, `FRA`
- **Usage:** All country codes
- **Reference:** <https://www.iso.org/iso-3166-country-codes.html>

#### E.164 Phone Numbers

- **Format:** `+` followed by country code and number
- **Example:** `+441632112233`, `+447700900123`
- **Validation:** `^\+[1-9][0-9]{1,14}$`
- **Usage:** All phone numbers

#### Email Addresses

- **Format:** Standard email format
- **Example:** `sarah.chen@example.com`
- **Validation:** RFC 5322 compliant
- **Usage:** All email addresses

### Identity and Reference Numbers

#### Task ID Format

- **Pattern:** `TASK-{TYPE}-YYYYMMDD-XXXXXX`
- **Types:** `VERIFY`, `SANC`, `RISK`, `COMP`, `ACCT`
- **Example:** `TASK-VERIFY-20250108-V7M3K1`
- **Usage:** Unique identifier for each task instance

#### Document ID Format

- **Pattern:** `DOC-YYYY-MMDD-XXX`
- **Example:** `DOC-2025-0108-001`
- **Usage:** Unique identifier for uploaded documents

#### Account Number Format

- **Pattern:** IBAN format where applicable
- **Example:** `GB29SLOC12345678901234`
- **Usage:** Client account identifiers

## Field Dependencies and Validation Rules

### Conditional Requirements

#### secondNationality

- **Required When:** Individual has dual citizenship
- **Required:** No
- **Parent Field:** `nationality`
- **Validation:** Must be different from primary nationality

#### tradingName

- **Required When:** Corporate entity trades under different name
- **Required:** No
- **Parent Field:** `legalName`

#### regulatorName

- **Required When:** `isRegulatedEntity` is true
- **Required:** Conditional
- **Parent Field:** `isRegulatedEntity`

#### stockExchange

- **Required When:** `isListedEntity` is true
- **Required:** Conditional
- **Parent Field:** `isListedEntity`

### Cross-Field Validations

#### Age Validation

- **Rule:** `dateOfBirth` must indicate individual is 18+ years old
- **Calculation:** Current date minus date of birth >= 18 years
- **Error:** "Applicant must be 18 years or older"

#### Document Expiry

- **Rule:** Identity documents must not be expired
- **Validation:** `expiryDate` must be >= current date
- **Exception:** Proof of address documents must be within 3 months

#### Ownership Percentage

- **Rule:** Total ownership percentage should typically equal 100%
- **Validation:** Sum of all `ownershipPercentage` values should be validated
- **Note:** May exceed 100% with complex structures (e.g., voting vs economic rights)

#### Beneficial Owner Threshold

- **Rule:** Individuals with >25% ownership must be disclosed
- **Validation:** All beneficial owners with `ownershipPercentage` > 25 must have complete details
- **Regulatory:** UK Money Laundering Regulations 2017

## Enumeration Values

### Complete Enumeration Definitions

#### clientType

- `individual`: Natural person
- `corporate`: Legal entity (company, partnership, trust, etc.)

#### employmentStatus

- `employed`: Full-time or part-time employment
- `self-employed`: Freelance, contractor, or own business
- `retired`: No longer working, receiving pension
- `unemployed`: Not currently employed
- `student`: In full-time education

#### legalForm

- `limited-company`: Private or public limited company
- `partnership`: General partnership
- `llp`: Limited liability partnership
- `trust`: Trust structure
- `foundation`: Foundation or stiftung
- `other`: Other legal forms (specify in notes)

#### assetClasses

- `equities`: Shares, stocks
- `fixed-income`: Bonds, gilts
- `fx`: Foreign exchange
- `commodities`: Physical or financial commodities
- `derivatives`: Options, futures, swaps
- `crypto`: Cryptocurrencies and digital assets

#### tradingFrequency

- `infrequent`: Less than quarterly
- `regular`: Weekly to monthly
- `high-frequency`: Daily or multiple times daily

#### verificationMethod

- `automated`: Fully automated verification
- `manual-review`: Human review required
- `hybrid`: Combination of automated and manual

#### sanctionType

- `asset-freeze`: Assets frozen, no business permitted
- `travel-ban`: Travel restrictions
- `arms-embargo`: Arms and military equipment embargo
- `financial-restrictions`: Restrictions on financial transactions
- `trade-restrictions`: General trade restrictions

#### matchType

- `exact`: Exact name and identifier match
- `close`: Very high confidence match (90%+)
- `possible`: Possible match requiring investigation (70-89%)

## Data Quality Standards

### Completeness Requirements

#### Mandatory Fields

All fields marked as "Required: Yes" must be populated. Null or empty values will cause validation failure.

#### Conditional Fields

Fields marked "Required: Conditional" must be populated when their condition is met.

#### Optional Fields

Fields marked "Required: No" may be omitted but are recommended for complete risk assessment.

### Data Accuracy

#### Name Matching

- Names must match across all documents and forms
- Minor variations (e.g., middle name inclusion/exclusion) are acceptable
- Significant discrepancies require explanation and remediation

#### Date Consistency

- Dates of birth must be consistent across all documents
- Document issue dates must be before expiry dates
- Incorporation dates must be before current date

#### Address Matching

- Addresses should match between application and proof of address
- Minor formatting differences (e.g., "Street" vs "St") are acceptable
- Significant discrepancies require remediation

### Data Freshness

#### Time-Sensitive Data

- Proof of address: Within 3 months
- Corporate registry documents: Within 3 months
- Certificate of incumbency: Within 3 months
- Board resolutions: Must be specific to current application

#### Static Data

- Identity documents: Must be valid (not expired)
- Articles of incorporation: Current version
- Memorandum and articles: Current version

## Data Retention and Privacy

### Retention Periods

#### Application Data

- **Approved Applications:** 7 years from account closure
- **Rejected Applications:** 7 years from rejection
- **Abandoned Applications:** 1 year from last activity

#### Document Storage

- **Identity Documents:** 7 years from account closure
- **Corporate Documents:** 7 years from account closure
- **Risk Assessments:** 7 years from creation
- **Audit Trails:** 7 years from process completion

### Data Protection

#### Personal Data

All personal data is subject to UK GDPR and Data Protection Act 2018.

#### Sensitive Personal Data

- Date of birth: Personal data
- National insurance/tax numbers: Sensitive personal data
- Biometric data: Special category data under GDPR

#### Right to Access

Individuals have the right to request access to their personal data (Subject Access Request).

#### Right to Rectification

Individuals can request correction of inaccurate data.

#### Data Minimisation

Only collect data necessary for KYC and regulatory compliance purposes.

### External System Data

#### CRM System

- **Receives:** clientId, client details, contact information

#### Document Management System

- **Receives:** Uploaded documents, document metadata

- **Sends:** Document images, client details

- **Receives:** All process events, decisions, timestamps

## Special Handling Requirements

### High-Risk Indicators

#### Automatic High-Risk Flags

- PEP match identified
- High-risk jurisdiction (score >75)
- Sanctions screening match (any)
- Adverse media (severity: high or critical)
- Complex ownership structure (>5 beneficial owners)
- Newly incorporated entity (<12 months)

#### Enhanced Due Diligence Triggers

- Any high-risk flag
- Risk score >50
- Trading volume >Â£10,000,000 annual
- Commodities or crypto trading
- Multiple jurisdictions involved

### Data Anonymisation

#### Test Data

All test and example data must use:

- Fictional names
- Example.com email domains
- Synthetic addresses
- Invalid but format-compliant identifiers

#### Demonstration Data

No real client data should ever be used in demonstrations or training materials.

## Version Control

### Schema Versioning

All schemas include a version number. Breaking changes require major version increment.

### Backward Compatibility

New fields may be added with minor version increments. Existing fields should not be removed or have their type changed without major version increment.

### Migration Path

When schemas change, migration scripts must be provided to convert existing data to new format.
